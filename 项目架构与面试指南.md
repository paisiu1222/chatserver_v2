# 集群聊天服务器 — 项目架构与面试指南

## 一、项目概述

基于 **Muduo 网络库** 的 **C++ 集群聊天服务器**，支持部署在 **Nginx TCP 负载均衡** 环境中。

核心功能：
- 用户注册与登录
- 一对一聊天（在线实时 + 离线存储）
- 好友管理（添加好友、好友列表）
- 群组管理（创建群组、加入群组、群聊）
- 通过 **Redis 发布-订阅** 实现跨服务器消息路由
- **MySQL** 做数据持久化

技术栈：C++11、Muduo、Redis (hiredis)、MySQL C API、nlohmann/json、CMake

### 层1：网络层 — ChatServer

`include/server/chatserver.hpp` + `src/server/chatserver.cpp`

```
ChatServer
├── TcpServer _server    ← Muduo 的 TcpServer，处理 TCP 连接
├── EventLoop *_loop     ← 事件循环指针
├── onConnection()       ← 连接建立/断开回调
└── onMessage()          ← 数据到达回调
```

关键实现细节：
- 构造函数中注册连接回调和消息回调（使用 `std::bind` 绑定成员函数）
- `setThreadNum(4)` — 4 个 I/O 工作线程（1 个 main reactor + 3 个 sub reactor）
- `onMessage` 中做 JSON 反序列化，提取 `msgid`，通过 `ChatService::getHandler()` 获取对应业务处理函数
- **网络层完全不知道业务逻辑细节，实现了解耦**

### 层2：业务层 — ChatService（核心）

`include/server/chatservice.hpp` + `src/server/chatservice.cpp`

```
ChatService (单例模式)
├── _msgHandlerMap:    unordered_map<int, MsgHandler>  ← 消息分发表
├── _userConnMap:      unordered_map<int, TcpConnectionPtr>  ← 在线用户连接池
├── _connMutex:        mutex  ← 保护 _userConnMap 的线程安全
├── _userModel:        UserModel
├── _offlineMsgModel:  OfflineMsgModel
├── _friendModel:      FriendModel
├── _groupModel:       GroupModel
└── _redis:            Redis
```

**消息分发机制（策略模式/命令模式）**：

构造函数中预注册所有消息类型对应的 handler：

```
_msgHandlerMap = {
    {LOGIN_MSG,       bind(&ChatService::login, ...)},
    {LOGINOUT_MSG,    bind(&ChatService::loginout, ...)},
    {REG_MSG,         bind(&ChatService::reg, ...)},
    {ONE_CHAT_MSG,    bind(&ChatService::oneChat, ...)},
    {ADD_FRIEND_MSG,  bind(&ChatService::addFriend, ...)},
    {CREATE_GROUP_MSG,bind(&ChatService::createGroup, ...)},
    {ADD_GROUP_MSG,   bind(&ChatService::addGroup, ...)},
    {GROUP_CHAT_MSG,  bind(&ChatService::groupChat, ...)}
};
```

新增消息类型只需在 `public.hpp` 加枚举值、在构造函数注册 handler、实现处理函数 — **无需修改网络层代码**，符合开闭原则。

**支持的消息类型** (`include/public.hpp`)：

| msgid | 枚举值 | 含义 |
|-------|--------|------|
| 1 | LOGIN_MSG | 登录请求 |
| 2 | LOGIN_MSG_ACK | 登录响应 |
| 3 | LOGINOUT_MSG | 注销请求 |
| 4 | REG_MSG | 注册请求 |
| 5 | REG_MSG_ACK | 注册响应 |
| 6 | ONE_CHAT_MSG | 一对一聊天 |
| 7 | ADD_FRIEND_MSG | 添加好友 |
| 8 | CREATE_GROUP_MSG | 创建群组 |
| 9 | ADD_GROUP_MSG | 加入群组 |

### 层5：跨服务器通信 — Redis Pub/Sub

`include/server/redis/redis.hpp` + `src/server/redis/redis.cpp`

这是实现**集群部署**的关键组件：

```
Redis 封装
├── _publish_context     ← 用于发布消息的 redisContext
├── _subscribe_context   ← 用于订阅消息的 redisContext
├── observer_channel_message()  ← 独立线程，阻塞等待订阅消息
└── _notify_message_handler     ← 收到消息后的业务层回调
```

**为什么需要两个 redisContext？**

`SUBSCRIBE` 命令会使连接进入"订阅模式"，之后该连接只能接收订阅消息，不能再发 `PUBLISH` 等命令。所以必须分离 publish context 和 subscribe context。

**为什么 observer 需要独立线程？**

`redisGetReply` 在 subscribe 模式下是阻塞的 — 没有消息时一直等。如果放在主线程，会把整个 EventLoop 卡死。

## 四、消息流转完整链路

### 一对一聊天（用户A → 用户B）

```
客户端A ──TCP──→ ChatServer::onMessage()
                    │
                    ▼ (JSON 反序列化, 提取 msgid=6)
            ChatService::getHandler(6)
                    │
                    ▼
            ChatService::oneChat()
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
    B在线(本机)   B在线(跨服)   B离线
                    │
    _userConnMap   _redis.publish  _offlineMsgModel
    直接 send()    → Redis channel    .insert()
                    → observer_channel
                    → handleRedisSubscribe
                    → B的conn send()
```

### 群聊消息流程

```
用户A发送群消息
    │
    ▼
ChatService::groupChat()
    │
    ▼
GroupModel::queryGroupUsers(userid, groupid)  ← 查出群内所有其他成员的 userid
    │
    ▼
遍历所有成员 userid:
    ├── 本机在线 → 直接 send()
    ├── Redis 在线(不在本机) → redis.publish(channel=userid, msg)
    └── 离线 → offlineMsgModel.insert(userid, msg)
```

## 七、面试高频问题

### 架构与设计

**Q1: 项目的整体架构是怎样的？**

分层架构：
1. **网络层**（ChatServer）：Muduo TcpServer 封装，处理 TCP 连接和 IO
2. **业务层**（ChatService）：单例模式 + 消息分发表，所有业务逻辑入口
3. **数据层**（Model 类族）：手写简易 ORM，封装 MySQL CRUD
4. **跨服通信层**（Redis）：Pub/Sub 实现集群中不同服务器实例间的消息路由

网络层通过 `msgid → MsgHandler` 映射表与业务层解耦，网络层不感知任何业务细节。

---

**Q2: 为什么选择 Muduo？Muduo 的线程模型是怎样的？**

Muduo 是陈硕开发的成熟 Reactor 网络库，核心是 "one loop per thread" 模型：
- 一个线程运行一个 EventLoop（epoll 实例）
- Main Reactor（主 EventLoop）负责 accept 新连接
- Sub Reactor（工作 EventLoop，本项目设了 4 个线程）负责已建立连接的读写事件
- 新连接通过 round-robin 分发给 sub reactor

比手写 epoll 更稳定，封装了 non-blocking IO、事件分发、缓冲区管理等细节。

---

**Q3: 消息分发是怎么实现的？用了什么设计模式？**

策略模式（命令模式）。核心是一个 `unordered_map<int, MsgHandler>`：
- 构造函数中预注册所有 `msgid → handler` 的映射
- 收到消息时，`getHandler(msgid)` O(1) 找到对应的处理函数
- 新增消息类型只需：加枚举 → 注册 handler → 实现处理函数，不改网络层代码
- 体现了**开闭原则**（对扩展开放，对修改封闭）

---

**Q4: `_userConnMap` 怎么保证线程安全？**

使用 `std::mutex` + `std::lock_guard`。Muduo 保证同一个 TcpConnection 的回调在同一线程执行，但 `_userConnMap` 是跨连接的共享数据，多个 sub reactor 线程可能同时读写，所以必须加锁。

可优化点：当前是独占锁，读写都互斥。如果读远多于写，可以改用 `std::shared_mutex`（C++17）实现读写锁。

---

**Q5: 服务启动时做了什么？如何保证优雅退出？**

`main.cpp` 中：
1. 注册 `SIGINT` 信号处理器 → `resetHandler`
2. `resetHandler` 调用 `ChatService::reset()` → 把所有 online 用户状态批量改为 offline
3. 这样即使服务器 crash，下次启动时不会出现大量"假在线"用户

---

### Redis 与集群

**Q6: Redis 在这个项目中具体怎么用的？**

作为集群中不同 ChatServer 实例之间的**消息总线**：
- 每个用户登录成功后，`_redis.subscribe(userid)` — 订阅以自己 userid 命名的 channel
- 服务器A要给 userB 发消息，但 userB 连接在服务器B上：
  - 服务器A 调用 `_redis.publish(userB_id, message)`
  - 服务器B 的 `observer_channel_message()` 独立线程收到消息
  - 回调 `handleRedisSubscribeMessage()` → 通过 `_userConnMap` 找到 userB 的 TCP 连接 → `send()`
- 用户登出时 `_redis.unsubscribe(userid)` 取消订阅

---

**Q7: 为什么 Redis 订阅需要独立线程？为什么需要两个 redisContext？**

- **独立线程**：`redisGetReply` 在 SUBSCRIBE 模式下是阻塞等待的，如果放在 muduo 的 IO 线程里会把 EventLoop 卡死
- **两个 redisContext**：SUBSCRIBE 后连接进入"订阅模式"，该连接只能收消息，不能发 PUBLISH。所以 publish 和 subscribe 必须各用一个独立的 `redisContext`

---

**Q8: 如果用户同时在两台服务器上登录（重复登录），怎么处理？**

当前代码中，登录时会检查 MySQL 中用户的 state 字段。如果已是 "online"，返回错误 `errno=2`："this account is using"。同时在构造函数中还连接了 Redis 做了订阅处理。

---

**Q9: Nginx 在这个项目中怎么配置？**

Nginx stream 模块做 TCP 四层代理：

```nginx
stream {
    upstream chatserver {
        server 192.168.1.10:6000;
        server 192.168.1.11:6000;
        server 192.168.1.12:6000;
    }
    server {
        listen 8000;
        proxy_pass chatserver;
    }
}
```

客户端连 `Nginx_IP:8000`，Nginx 根据负载均衡策略（默认 round-robin）分发连接到不同的 ChatServer 实例。Redis pub/sub 保证分散在不同服务器的用户之间可以通信。

### 数据库

**Q10: 数据库连接池这里是怎么处理的？**

当前没有连接池。每次 CRUD 操作都 `new MySQL()` → `connect()` → 操作 → 析构 `close()`。每次都做 TCP 三次握手 + MySQL 认证，高并发下是性能瓶颈。

改进方向：使用连接池（如自己基于队列 + RAII 实现，或引入第三方库），或者至少改为长连接复用。

---

**Q11: 离线消息是怎么处理的？**

- 发送消息时，如果目标用户不在本机且 Redis 状态也不在线 → 判定为离线 → `_offlineMsgModel.insert(userid, json_msg)`
- 用户登录成功时，`_offlineMsgModel.query(userid)` 拉取所有离线消息，放在登录响应的 `offlinemsg` 字段里返回给客户端
- 拉取后 `_offlineMsgModel.remove(userid)` 删除，防止重复拉取

**Q12: 用户密码是怎么存储的？**

当前是**明文存储和明文比较**：`user.getPwd() == pwd`。这是安全问题。

改进方向：注册时加盐哈希存储（如 bcrypt），登录时对输入密码做同样哈希后比对。

**Q13: 数据库操作有 SQL 注入风险吗？**

有。当前使用 `sprintf` 直接拼接 SQL，例如用户名含单引号会破坏 SQL 语法。应使用参数化查询（MySQL prepared statements: `mysql_stmt_bind_param`）。

---

### 并发相关

**Q14: Muduo 的 onMessage 回调在哪个线程执行？**

在 sub reactor 的 EventLoop 线程中执行。Muduo 保证**同一个 TcpConnection 的所有事件回调都在同一个 IO 线程**中，所以同一连接不会并发。但不同连接可能在不同线程，因此跨连接的共享数据（如 `_userConnMap`）需要加锁。

**Q15: `ChatService` 单例的线程安全性？**

采用 C++11 的 **Meyers' Singleton**：
```cpp
static ChatService service;
return &service;
```
C++11 保证 static 局部变量的初始化是线程安全的（编译器自动加锁）。

---

### 客户端相关

**Q16: 客户端为什么用信号量做同步？**

主线程发登录请求后必须等子线程收到响应才能继续。用 `sem_wait/sem_post` 实现，避免了轮询（busy wait）。更现代化的做法可以用 `std::promise/std::future` 或 `std::condition_variable`。

**Q17: 客户端用原生 socket 而不用 Muduo？**

客户端只需一个 TCP 连接，逻辑简单。原生 socket 足够且不引入额外依赖。服务端需要处理大量并发连接，才需要 Muduo 的 Reactor 模型。

### 可改进方向（主动提出体现深度思考）

1. **连接池**：MySQL 和 Redis 连接复用，减少建连开销
2. **密码安全**：加盐哈希（bcrypt/argon2）
3. **SQL 注入**：使用 prepared statements
4. **用户状态**：online/offline 迁移到 Redis，减轻 MySQL 压力
5. **序列化**：JSON → Protobuf/FlatBuffers，减小包体、提高解析速度
6. **读写锁**：`_userConnMap` 用 `shared_mutex` 优化读多写少场景
7. **心跳机制**：检测死连接，及时清理
8. **消息可靠性**：ACK 机制 + 消息去重，防止消息丢失或重复投递
9. **限流/熔断**：防止恶意客户端洪水攻击
10. **日志系统**：当前日志较简单，可引入结构化日志 + 日志级别控制

    json js = json::parse(buf);  // 反序列化
    auto msgHandler = ChatService::instance()->getHandler(js["msgid"].get<int>());

### 一对一聊天：三级路由策略

```cpp
void ChatService::oneChat(const TcpConnectionPtr &conn, json &js, Timestamp time)
{
    int toid = js["toid"].get<int>();
    {
        lock_guard<mutex> lock(_connMutex);
        auto it = _userConnMap.find(toid);
        if (it != _userConnMap.end()) {
            it->second->send(js.dump());  // 本机在线 → 直接转发
            return;
        }
    }
    User user = _userModel.query(toid);
    if (user.getState() == "online") {
        _redis.publish(toid, js.dump());  // 跨服务器在线 → Redis 转发
        return;
    }
    _offlineMsgModel.insert(toid, js.dump());  // 离线 → 存离线消息
}
```
