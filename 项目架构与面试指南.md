## 一、项目概述

基于 **Muduo 网络库** 的 **C++ 集群聊天服务器**，支持 Nginx TCP 负载均衡环境下的多实例部署，通过 Redis Pub/Sub 实现跨服务器消息路由。

**核心功能：**
- 用户注册与登录（PBKDF2-HMAC-SHA256 密码哈希）
- 一对一聊天（在线实时转发 + 离线消息存储）
- 好友管理（添加好友、好友列表）
- 群组管理（创建群组、加入群组、群聊）
- 心跳检测（空闲连接自动清理）
- 消息去重与 ACK 确认
- MySQL 连接池、Redis 在线状态缓存

**技术栈：** C++11/17 | Muduo | Redis (hiredis) | MySQL C API | nlohmann/json | OpenSSL | CMake | Docker

## 四、消息流转链路

### 4.1 一对一聊天（用户A → 用户B）

```
ClientA ──TCP──→ ChatServer::onMessage()
                    │ JSON 反序列化, msgid=6
                    ▼
            ChatService::oneChat()
                    │
            ┌───────┤ msg_uuid 去重检查 (Redis SETNX)
            │       │
            │       ├── 重复 → 仅回复 ACK "duplicate"
            │       │
            │       ▼ 首次消息
            │   ┌───────────┬───────────┐
            │   ▼           ▼           ▼
            │ B在线(本机)  B在线(跨服)   B离线
            │ _userConnMap  Redis GET    MySQL
            │ 直接 send()  publish()    insert()
            │
            └──→ 回复 MSG_ACK {status: "ok"/"offline"}
```

### 4.2 群聊流程

```
ClientA → groupChat()
    │ 去重检查
    ├── queryGroupUsers() 获取群内其它成员
    └── 遍历每个成员:
        ├── 本机在线 → 直接 send()
        ├── Redis 在线 → redis.publish(channel=userid, msg)
        └── 离线 → offlineMsgModel.insert()
    → 回复 MSG_ACK
```

### 4.3 心跳流程

```
服务端：
  main() → loop.runEvery(30s) → checkIdleConnections()
    ├── 遍历 _lastActiveMap
    ├── 超过 90s 无活动 → 断开连接 + 清理状态
    └── _lastActiveMap 由 login/chat/groupChat/PING 更新

客户端：
  任何消息（含 PING）都更新服务端活跃时间
  收到 PONG_MSG 静默忽略
```

## 八、面试高频问题（v2.0 更新版）

### 架构与设计

**Q1: 描述项目的整体架构。**

五层架构：
1. **网络层**（ChatServer，Muduo TcpServer + 4 线程 Reactor）
2. **业务层**（ChatService 单例，策略模式消息分发）
3. **数据层**（简易 ORM，Model 类族封装 MySQL CRUD）
4. **基础设施层**（连接池、Redis 多用途封装、PBKDF2 密码模块、配置管理）
5. **跨服通信层**（Redis Pub/Sub 消息总线）

网络层与业务层通过 `msgid → MsgHandler` 查表回调完全解耦。

---

**Q2: Muduo 的线程模型？为什么选 Muduo？**

"one loop per thread"：主 Reactor 负责 accept，通过 round-robin 分发给 3 个 sub Reactor 处理已建立连接的读写。Muduo 封装了 epoll + non-blocking IO + 缓冲区管理，比手写 epoll 更稳定可靠。每个 TcpConnection 的所有回调保证在同一线程执行，大幅简化了并发编程模型。

---

**Q3: 消息分发用什么设计模式？优点是什么？**

策略模式/命令模式。核心是 `unordered_map<int, function<void(conn, json, time)>>`：
- 构造函数预注册所有 `msgid → handler`
- 运行时 `getHandler(msgid)` O(1) 查找
- 新增消息类型（如 PING/PONG/ACK）只需：加枚举 → 注册 → 实现，不需改网络层代码
- 符合开闭原则

---

**Q4: `_userConnMap` 的线程安全怎么保证？**

使用 `std::shared_mutex`（读写锁）：
- **读操作**（oneChat/groupChat 查 map、handleRedisSubscribeMessage）：`shared_lock`，允许多线程并发读
- **写操作**（login 插入、loginout/clientCloseException 删除）：`unique_lock`，独占写

Muduo 保证同一连接的回调在同一线程，但不同连接可能在不同 sub reactor 线程，共享数据需要锁保护。读写锁比原来的 `mutex`（所有操作互斥）更适合读多写少场景。

---

**Q5: 连接池是怎么实现的？为什么需要它？**

`ConnectionPool` 单例，启动时预建 N 个连接放入 `queue<MYSQL*>`：
- `acquire()`：`unique_lock` + `condition_variable::wait`，队列空时阻塞等待
- `release()`：归还连接并 `notify_one` 唤醒等待者
- MySQL 类的 `connect()` 调用 `acquire()`，析构函数调用 `release()`
- RAII 自动管理，Model 层代码无需改动

每次新建连接需要 TCP 三次握手 + MySQL 认证（~10ms），连接池消除了这个开销。

---

### Redis 与集群

**Q6: Redis 在这个项目中承担了几种角色？**

三种：
1. **Pub/Sub 消息总线**：跨服务器消息路由
2. **K-V 状态缓存**：用户 online/offline 状态（减少 MySQL 查询）
3. **消息去重**：`SET msg:uuid 1 NX EX 3600` 防止重复投递

---

**Q7: 为什么 Redis 需要两个 redisContext 和独立线程？**

- **两个 context**：SUBSCRIBE 后连接进入"订阅模式"，只能收消息不能发 PUBLISH，必须分离
- **独立线程**：`redisGetReply` 在 SUBSCRIBE 模式阻塞等待，放主线程会卡死 EventLoop
- publish_context 加了 `mutex` 保护（多线程并发调用 PUBLISH/SET/GET）

---

**Q8: 用户在线状态为什么用 Redis 缓存而不是每次都查 MySQL？**

原有设计每次发消息都 `_userModel.query(toid)` 查 MySQL 判断对方是否在线。改为：
1. 登录时 `SET user:1:state online`
2. 发消息时先 `GET user:{toid}:state` (Redis O(1) μs 级)
3. 未命中才回源 MySQL 并回写 Redis
4. 登出/异常断开时 `DEL user:1:state`

减少 MySQL 高频查询压力，消息路由延迟从 ms 降至 μs。

---

**Q9: 心跳机制是怎么实现的？**

- 服务端维护 `_lastActiveMap<userid, Timestamp>`，在 login/chat/groupChat/PING 时更新
- `EventLoop::runEvery(30s)` 定时调用 `checkIdleConnections()`
- 超过 90 秒无活动 → `shutdown()` 连接 + 清理 `_userConnMap` + Redis 状态 + MySQL 状态
- 客户端无感知（MSG_ACK/PONG 等响应静默处理）

---

**Q10: 消息去重和 ACK 是怎么做的？**

- 每条消息携带 `msg_uuid`（`userid_timestamp`）
- 服务端用 `SET msg:{uuid} 1 NX EX 3600` 判断是否重复
- 首次消息：处理 + 回复 `MSG_ACK {status: "ok"/"offline"}`
- 重复消息：仅回复 `MSG_ACK {status: "duplicate"}`
- 客户端可基于 ACK 实现超时重发

---

### 密码安全

**Q11: 密码怎么存储的？为什么这样设计？**

PBKDF2-HMAC-SHA256：
- 随机生成 16 字节 salt
- 100000 次迭代派生 32 字节 key
- 存储格式：`hex(salt):hex(key)` = 97 字符
- 验证时用 `CRYPTO_memcmp` 常量时间比较，防时序侧信道攻击

选择 PBKDF2 而非 bcrypt 的原因：OpenSSL 内置支持，无需额外依赖，业界公认安全。

---

### 数据库

**Q12: SQL 注入怎么防的？**

- 所有 `sprintf` 替换为 `snprintf`，防止缓冲区溢出
- 字符串参数通过 `mysql_real_escape_string` 转义单引号、反斜杠等特殊字符
- 输入校验层：用户名/密码/消息长度在业务层先行过滤
- 静态 SQL（如 `resetState()`）无用户输入，直接使用常量字符串

---

**Q13: 服务如何优雅退出？**

1. `SIGINT` 信号 → `resetHandler`
2. `ChatService::reset()` → `UPDATE user SET state='offline' WHERE state='online'`
3. Redis 中的状态数据设了 TTL 或随登出清理
4. 连接池在静态析构时释放所有连接

---

### 并发与 C++

**Q14: `ChatService` 单例的线程安全性？**

Meyers' Singleton：C++11 保证 `static` 局部变量初始化线程安全（编译器自动加双重检查锁）。

**Q15: `shared_mutex` vs `mutex` 的使用场景？**

- `shared_mutex`：读多写少（`_userConnMap` 90% 读操作）
- `mutex`：写多或临界区极小（`_ctxMutex` 保护 Redis 命令）
- 选择依据：profiling 观察锁竞争情况

**Q16: 项目中哪里体现了 RAII？**

- MySQL 类：析构自动归还连接到池
- `lock_guard` / `unique_lock` / `shared_lock`：自动释放锁
- `ConnectionPool`：析构时关闭所有连接

### 部署

**Q17: 如何用 Docker 部署？**

```bash

```

启动 5 个容器：MySQL + Redis + ChatServer × 2 + Nginx。MySQL 初始化脚本自动建表，Nginx stream 模块做四层负载均衡，客户端连 `localhost:8000` 即可。

### 一对一聊天：三级路由 + 去重 + ACK

```cpp
// chatservice.cpp
void ChatService::oneChat(const TcpConnectionPtr &conn, json &js, Timestamp time)
{
    int toid = js["toid"].get<int>();
    string msg = js.value("msg", "");
    if (toid <= 0 || !isValidMessage(msg)) return;

    // 去重检查
    string msgUuid = js.value("msg_uuid", "");
    if (!msgUuid.empty() && !_redis.setnxWithExpire("msg:" + msgUuid))
    {
        json ack; ack["msgid"] = MSG_ACK; ack["status"] = "duplicate";
        conn->send(ack.dump()); return;
    }

    // 更新活跃时间
    { unique_lock<shared_mutex> lock(_connMutex);
      _lastActiveMap[js["id"].get<int>()] = time; }

    // 本机在线 → 直接转发
    { shared_lock<shared_mutex> lock(_connMutex);
      auto it = _userConnMap.find(toid);
      if (it != _userConnMap.end()) {
          it->second->send(js.dump());
          json ack; ack["msgid"] = MSG_ACK; ack["status"] = "ok";
          conn->send(ack.dump()); return;
      }}

    // Redis 缓存查状态，未命中回源 MySQL
    string toState = _redis.getUserState(toid);
    if (toState.empty()) {
        User user = _userModel.query(toid);
        toState = user.getState();
        if (toState == "online") _redis.setUserState(toid, "online");
    }
    if (toState == "online") {
        _redis.publish(toid, js.dump());
    } else {
        _offlineMsgModel.insert(toid, js.dump());
    }
    json ack; ack["msgid"] = MSG_ACK; ack["status"] = toState == "online" ? "ok" : "offline";
    conn->send(ack.dump());
}
```

### 心跳检测

```cpp
// main.cpp
loop.runEvery(30.0, []() {
    ChatService::instance()->checkIdleConnections();
});

// chatservice.cpp
void ChatService::checkIdleConnections()
{
    Timestamp now = Timestamp::now();
    vector<int> staleUsers;
    {
        unique_lock<shared_mutex> lock(_connMutex);
        for (auto it = _lastActiveMap.begin(); it != _lastActiveMap.end();)
        {
            if (timeDifference(now, it->second) > 90.0)
            { staleUsers.push_back(it->first); it = _lastActiveMap.erase(it); }
            else ++it;
        }
    }
    for (int userid : staleUsers)
    {
        { unique_lock<shared_mutex> lock(_connMutex);
          auto it = _userConnMap.find(userid);
          if (it != _userConnMap.end()) { it->second->shutdown(); _userConnMap.erase(it); } }
        _redis.unsubscribe(userid);
        _redis.delUserState(userid);
        _userModel.updateState(User(userid, "", "", "offline"));
    }
}
```

## 十二、总结

该项目是一个**生产环境入门级的 C++ 分布式即时通讯系统**，涵盖后端核心能力：

- **网络编程**：Muduo Reactor 模型、TCP 长连接管理、Nginx 四层负载均衡
- **系统设计**：五层分层架构、策略模式/单例模式/RAII、开闭原则
- **分布式通信**：Redis Pub/Sub 跨节点消息路由、状态缓存、消息去重
- **安全编码**：PBKDF2 密码哈希、SQL 注入防护、输入校验、常量时间比较
- **并发控制**：shared_mutex 读写锁、条件变量、Meyers' Singleton
- **工程化**：Docker 容器化、配置管理、优雅退出、连接池
