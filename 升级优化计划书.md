## 三、分阶段实施计划

### 第一阶段：安全加固（预计 1-2 天）

#### 任务 1.1：密码加盐哈希

**涉及文件**：
- `src/server/model/usermodel.cpp` — 注册时哈希存储
- `src/server/chatservice.cpp` — 登录时哈希比对

**技术方案**：
- 引入 bcrypt 库（如 `libbcrypt` 或 `openssl` 内置）
- 注册：`hash = bcrypt(password, salt)`，将 hash 存入数据库
- 登录：从数据库取出 hash，`bcrypt_verify(input_password, hash)`
- 数据库 `password` 字段长度需扩展至 60 字符（bcrypt hash 长度）

**验收标准**：
- 数据库中不再存储明文密码
- 登录功能正常
- 旧明文密码用户不可用（强制重新注册或做数据迁移）

---

#### 任务 1.2：SQL 注入防护（参数化查询）

**涉及文件**：
- `src/server/model/usermodel.cpp`
- `src/server/model/friendmoel.cpp`
- `src/server/model/groupmodel.cpp`
- `src/server/model/offlinemessagemodel.cpp`

**技术方案**：
- 将所有 `sprintf(sql, ...)` 拼接替换为 MySQL Prepared Statements
- 使用 `mysql_stmt_init()` → `mysql_stmt_prepare()` → `mysql_stmt_bind_param()` → `mysql_stmt_execute()`
- 字符串参数用 `MYSQL_TYPE_STRING` 绑定，整数用 `MYSQL_TYPE_LONG`

**验收标准**：
- 包含单引号的用户名/消息不再导致 SQL 错误
- 所有 CRUD 功能回归正常

---

#### 任务 1.3：输入校验

**涉及文件**：
- `src/server/chatservice.cpp` — 各 handler 入口
- `include/public.hpp` — 添加校验常量

**技术方案**：
- 用户名：长度 2-20，只允许字母数字下划线中文
- 密码：长度 6-100
- 消息内容：长度 1-5000
- 群名/群描述：长度 1-50 / 1-200
- 校验不通过直接返回错误，不执行业务逻辑

**验收标准**：
- 超长/非法输入被拒绝，返回明确错误信息
- 正常输入不受影响

### 第二阶段：性能优化（预计 2-3 天）

#### 任务 2.1：MySQL 连接池

**涉及文件**：
- `include/server/db/connectionpool.hpp` — 新增
- `src/server/db/connectionpool.cpp` — 新增
- `include/server/db/db.h` — 修改 MySQL 类使用连接池
- `src/server/db/db.cpp` — 同上
- 所有 Model `.cpp` 文件 — 改为从池中获取连接

**技术方案**：
```cpp
class ConnectionPool {
public:
    static ConnectionPool* instance();  // 单例
    MYSQL* getConnection();             // 获取连接（池空则阻塞等待）
    void releaseConnection(MYSQL*);     // 归还连接
private:
    queue<MYSQL*> _connQueue;
    mutex _mutex;
    condition_variable _cv;
    int _poolSize;  // 默认 8
};
```
- 使用 RAII 包装类 `ConnectionGuard`，构造时获取连接，析构时归还
- 启动时预创建 N 个连接

**验收标准**：
- 1000 并发请求下不再出现连接耗尽
- 数据库操作延迟降低 50%+（减少建连开销）

---

#### 任务 2.2：用户在线状态迁移到 Redis

**涉及文件**：
- `src/server/model/usermodel.cpp` — 登录/登出时同步写 Redis
- `src/server/chatservice.cpp` — 查状态改用 Redis GET
- `src/server/redis/redis.cpp` — 添加 `setKey`/`getValue` 方法
- `include/server/redis/redis.hpp` — 同上

**技术方案**：
- Redis Key 格式：`user:{id}:state`，Value：`"online"` / `"offline"`
- 登录时 `SET user:{id}:state online`
- 登出时 `SET user:{id}:state offline`
- 发消息时先 `GET user:{toid}:state`，命中则直接判断；未命中则回源 MySQL 并回写 Redis
- 服务启动时 `ChatService::reset()` 中批量将 Redis online 用户设回 offline

**验收标准**：
- 消息路由时不再查 MySQL
- 用户状态查询延迟从 ms 级降至 us 级

---

#### 任务 2.3：_userConnMap 读写锁优化

**涉及文件**：
- `include/server/chatservice.hpp` — `mutex` → `shared_mutex`
- `src/server/chatservice.cpp` — 所有 `lock_guard` → `shared_lock`（读）/ `unique_lock`（写）

**技术方案**：
- 读操作（getHandler、oneChat 查 map、groupChat 查 map）：`std::shared_lock`
- 写操作（login 插入、loginout 删除、clientCloseException 删除）：`std::unique_lock`

**验收标准**：
- 编译通过（需 C++17 或 Boost）
- 高并发读取场景下锁竞争显著降低

### 第三阶段：可靠性增强（预计 2-3 天）

#### 任务 3.1：心跳检测机制

**涉及文件**：
- `include/public.hpp` — 新增 `PING_MSG` / `PONG_MSG` 消息类型
- `src/server/chatserver.cpp` — 服务端空闲超时检测
- `src/server/chatservice.cpp` — 心跳消息处理
- `src/client/main.cpp` — 客户端定时发送心跳

**技术方案**：
- 新增消息类型：`PING_MSG = 11`（客户端→服务端），`PONG_MSG = 12`（服务端→客户端）
- 服务端：记录每个连接的最后活跃时间，用 Muduo `EventLoop::runEvery()` 每 30 秒扫描一次，超过 90 秒无消息则断开
- 客户端：每 30 秒发送 PING，30 秒内未收到 PONG 则重连
- 断开逻辑复用现有 `clientCloseException()`

**验收标准**：
- 客户端 kill -9 后，服务端 90 秒内检测到并清理连接
- 用户状态正确变为 offline
- 正常连接不受影响

---

#### 任务 3.2：消息 ACK 与去重

**涉及文件**：
- `include/public.hpp` — 新增消息类型
- `src/server/chatservice.cpp` — ACK 回复 + 去重逻辑
- `src/client/main.cpp` — ACK 等待 + 超时重发
- `src/server/redis/redis.cpp` — SETNX 去重
- `include/server/redis/redis.hpp` — 同上

**技术方案**：
- 客户端每条消息携带 `msg_uuid`（UUID 或 时间戳+userid+序列号）
- 服务端收到后 `SETNX msg:{msg_uuid} 1 EX 3600`，已存在则忽略（去重）
- 处理后回复 `MSG_ACK {msg_uuid, status}`
- 客户端发消息后启动 5 秒定时器，未收到 ACK 则重发（最多 3 次）
- 离线消息按 `seq` 字段排序，客户端启动时从上次 seq+1 开始拉取

**验收标准**：
- 网络抖动场景下消息不丢失、不重复
- 重复发送同一条消息，服务端只处理一次

---

#### 任务 3.3：通信协议升级（JSON → Protobuf）

**涉及文件**：
- `proto/chat.proto` — 新增，定义所有消息格式
- `src/server/chatserver.cpp` — 序列化方式切换
- `src/server/chatservice.cpp` — 同上
- `src/client/main.cpp` — 同上
- `CMakeLists.txt` — 添加 protobuf 编译

**技术方案**：
```protobuf
syntax = "proto3";
message ChatMessage {
    int32 msgid = 1;
    int32 id = 2;
    string name = 3;
    int32 toid = 4;
    string msg = 5;
    string time = 6;
    // ...
}
```
- 先用 protoc 生成 C++ 代码
- 网络传输用 `SerializeToArray()` / `ParseFromArray()`
- 保留 JSON 作为可选协议（通过 msgid 第一位区分）

**验收标准**：
- 消息包大小减少 60%+
- 序列化/反序列化速度提升 5x+
- 前后兼容性（新增字段不影响旧客户端）

## 四、实施顺序与依赖关系

```
第一阶段（安全）          第二阶段（性能）          第三阶段（可靠性）        第四阶段（工程化）
                                                                
bcrypt 哈希 ─────┐                                              配置文件 ──────┐
                 │                                                              │
参数化查询 ──────┤                                              日志升级 ──────┤
                 │                                                              │
输入校验 ────────┘                                              单元测试 ──────┤
     │                                                                          │
     └─────→ 连接池 ──────→ 心跳机制 ──────→ Docker 化 ─────────┘
                 │              │
            状态缓存        ACK + 去重
                 │              │
            读写锁优化    Protobuf 协议
                 │              │
                 └──────┬───────┘
                        │
                    编译 + 回归测试
```

- 安全修复必须在所有其他改动之前（保证后续改动建立在安全基础上）
- 连接池是状态缓存和心跳机制的前置依赖（Model 层重构影响面大）
- Docker 化放在最后（依赖所有改动稳定后统一打包）

## 五、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| bcrypt 引入编译困难 | 阻塞任务 1.1 | 备选：SHA256 + 固定盐（虽不如 bcrypt 但远好于明文） |
| Prepared Statement 改动大 | Model 层全部重写 | 先写一个 Model 验证方案，再批量迁移 |
| protobuf 导致前后不兼容 | 新旧协议无法互通 | 保留 JSON 协议，通过 msgid 高位区分 |
| 连接池引入死锁 | 服务不可用 | 设置获取连接超时（5秒），超时后告警 + fallback 新建连接 |
| Docker 环境搭建耗时 | 阻塞任务 4.3 | 分步验证：先确保单机编译运行，再容器化 |
| C++17 特性兼容性 | 编译失败 | 读写锁用 boost::shared_mutex 替代 std::shared_mutex |
